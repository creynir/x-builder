**Carried P2 concerns (from delta validation) to honor in the relevant tickets:** (a) a per-chain LLM **timeout/budget** + clean failure surfacing for the 3-call apply chain and 4-call generate-refine (XOB-011, XOB-012); (b) **edit-while-applying cancellation** for the apply chain (XOB-027); (c) **rect-thrash visual budget** during rapid typing (XOB-022 Visual AC).

**Carried from XOB-016 → XOB-030 [INT] (the real `BoundEngineServices` adapter + RunnerApp wiring):** the `ExposeFunctionTransport` binder is structurally decoupled (shared types only), so XOB-030 owns building the real adapter bundle. It MUST: (1) map `judgeDraft` `req → JudgeDraftService.judge(req.text, req.accountProfile)` and `JudgeDraftOutcome → JudgeDraftResponse`; (2) **re-attach per-item `cooldown` in the `analyzePosts` adapter** (resolver-chain → `DeterministicAnalysisService.analyzePosts` → `RepetitionWindowService.compute(windowDays)` cooldown-attach, mirroring `server.ts` `attachCooldownSignals`) **and assert it on round-trip** — `cooldown` is schema-optional so it silently vanishes otherwise, breaking XOB-025's per-item cooldown UX; (3) wire RunnerApp `bindTransport` default → `ExposeFunctionTransport.bindAll` + construct the full bundle incl. `getStatus` composition (no `AppStatusService` class — compose from the engine `/status` logic) and `getOverlayReadiness` (from XOB-017); (4) **wire RunnerApp `attachObserver` default → `GraphQlCaptureObserver.attach(context, batch => liveCaptureService.ingest(batch))`** and register `getOverlayReadiness(<engine readiness wrapped to a `getSubsystems()` shape — the engine `ReadinessService` exposes `getStatus()`, not `getSubsystems()`>, observer)` into the bundle. Capture is **inert end-to-end until this wiring lands** — XOB-030 must prove capture→corpus ingest + the readiness round-trip in-process.

**Carried from XOB-019 → XOB-029 (AnchorLayer pin lifecycle API + ComposeContext + compose detection) — REDIRECTED from XOB-025 (2026-06-22):** XOB-019's `AnchorLayer` ships as a read-only/empty registry skeleton (`Map<Element, AffordanceHandle>` + `useAnchorRegistry()`). The **pin register/reconcile mutation API** (pins mount/unmount on DOM churn; a composer-keyed `ComposeContext` entry `{composerEl, isActive, composerText}`), the compose detection (URL `/compose/post` OR `[role="dialog"]`+`tweetTextarea_0`), the 350 ms debounced `.textContent` read, and the `analyzePosts` trigger are unbuilt. **Originally slated for XOB-025, these are now owned by XOB-029 (ComposeCockpit assembly)** — XOB-025's `StaticEngineColumn` family turned out to be purely presentational (driven by an injected `analyzeState` prop; its 8 tests exercise no detection), so building an under-tested AnchorLayer/detection API there was rejected. XOB-029 mounts the 3 zone pins through the additively-extended `AnchorLayer`, derives `ComposeContext`, runs the `analyzeState` machine (fires `analyzePosts` on debounced text + `getCaptureSummary().followers` once on context open), and feeds `ComposeGenerateRail`/`StaticEngineColumn`/`JudgeStrip`; XOB-030 proves no-orphans on SPA nav. The seam is extended additively, not re-opened. (Also: `getSettings()` returns the `AppSettingsResponse` envelope — XOB-020 unwraps `.settings`. And XOB-030 references `overlay/src/anchor/anchor-layer.ts` while XOB-019 shipped flat `overlay/src/anchor-layer.tsx` — reconcile the import path at XOB-030.)

**Carried from XOB-018 → XOB-019 (overlay token substrate): RESOLVED in XOB-019** — the overlay shadow root seeds only the 25 `--xb-*` tokens. Downstream components also consume `--space-*`/`--type-*`/`--radius-*`/`--score-*` (defined `:root`-scoped in `docs/design-system/product-tokens.css`), which **do not resolve inside the shadow root** because the overlay is isolated and x.com has no `:root` design tokens. **XOB-019 must seed the consumed design-system primitive tokens onto the shadow `:host`** (extend the neon sheet or add a base-tokens sheet). First bites at XOB-021 (`--type-caption`/`--radius-md`), XOB-024 (`--space-2`), XOB-025 (`--score-*`). The overlay test harness is **Vitest browser mode → Playwright Chromium** (established in XOB-018); later overlay tickets inherit it.

**Overlay v2 primitive library (wave-wide, user-directed at XOB-020):** the existing `client/src/ui/foundation.tsx` primitives are NOT reused (they don't fit the new overlay UX). A fresh **v2 component library** is built **incrementally in `client/src/ui/v2/`** (beside the old `foundation.tsx`), per `docs/design-system/product-components.md`, **token-driven + self-contained/shadow-DOM-portable** (styles injected into the overlay shadow root, not global CSS). Overlay affordances (`overlay/src/<feature>/`) consume the v2 primitives cross-package. First-consumer builds each primitive: XOB-020 → Button/IconButton/Input/Alert/Badge/Switch/Skeleton/KeyValueList; XOB-025 → ScoreBar; etc. (Possible future cleanup: extract v2 to a shared package if the overlay→client import coupling becomes awkward.)

**Carried from XOB-022 → XOB-029 (single per-frame rect snapshot):** XOB-022's `CompositionHighlightLayer` reads its shadow-host origin (`getLayerOrigin`) and the composer box (`useComposerRect`) on **independently-debounced ticks**, so during modal motion the two can momentarily disagree → transient horizontal jitter (self-healing within ≤1 frame). **XOB-029 (`ComposeCockpit` assembly) must feed one per-frame snapshot** — host origin + composer rect captured together through the shared rAF rect tracker — to the highlight layer. This is the correct shape for any origin model and is required regardless of the XOB-022 L1 dual-origin cleanup. (XOB-022 L1 — collapse the host-vertical/composer-horizontal dual origin to a single host origin + recalibrate Red's `span.style.left < composerRect.left` test-artifact assertion — is an **internal, non-contract XOB-022 follow-up**, deferred per the `[FND]` checkpoint; it does not block any downstream ticket.) **Also fold in XOB-023 C1:** `useComposerText` commits its debounced read with `flushSync`, so once `ProvenanceController` is mounted in `ComposeCockpit`, a forced synchronous render on the debounce tick will flush any descendant highlight re-map in the same frame — capturing host origin + composer rect + composer text in one per-frame rAF snapshot resolves XOB-022 L2 and this XOB-023 interaction together. (XOB-023 C1 also carries an internal test-observation cleanup — drop `flushSync` for plain `setState` + `act`-aware test settling — that is non-blocking and can ride along here.)

**Carried from XOB-024 → XOB-025/026/028 (v2 `Button` has no width/`block` prop): D1.** v2 `Button` is `display:inline-flex` and exposes no `width`/`block`/`fullWidth`/`style`/`className` prop, so a consumer cannot explicitly request a full-width pill. XOB-024 satisfied its "full-width pill list" layout implicitly via a `flex-direction:column` container's default `align-items:stretch`. Upcoming zone components that mount v2 `Button` in pinned/stacked layouts (XOB-025 `StaticEngineColumn`/Post-Coach, XOB-026 `JudgeStrip`, XOB-028 `SuggestAffordance`) should do the same (rely on container `align-items:stretch`, or set it explicitly for clarity) — and must NOT fork the primitive via a style/className escape hatch (Visual-AC "don't fork the primitive"). If a third consumer genuinely needs explicit width control, that is the rule-of-three trigger to add a `block`/`fullWidth` prop to v2 `Button`.
